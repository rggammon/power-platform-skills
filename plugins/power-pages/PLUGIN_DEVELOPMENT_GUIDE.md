# Plugin UX & Reliability

## Why This Matters

A plugin can have 18 brilliant skills and still feel broken if the user sits staring at a frozen terminal for 90 seconds, has no way to know if the schema actually deployed, or loses their main conversation context because a long deployment blocked the thread.

Here are **five UX and reliability pillars** that as a quality standard every skill and command must meet before it ships.

---

## The Five Pillars

These five engineering principles address the gap between "the plugin works" and "the plugin feels great to use." Each pillar is a non-negotiable requirement embedded into the acceptance criteria for every skill and command.

### 01 — Engaging Loading Experience

**Pillar 1 — Perception of speed**

Long operations must show structured progress — not silence. The user should always know what's happening, what stage they're in, and roughly how long remains.

### 02 — Action Verification

**Pillar 2 — Trust through proof**

Every action the LLM performs must be verifiable. The agent should confirm what it did, show evidence it worked, and give the user a way to validate independently.

### 03 — Sub-Agent Architecture

**Pillar 3 — Context preservation**

Long-running processes (deployments, schema pushes, test suites) should run as sub-agents in isolated contexts, keeping the main conversation responsive and clean.

### 04 — PAC CLI Determinism

**Pillar 4 — Reliability through determinism**

Deterministic operations (upload, download, deploy, list) should be routed through PAC CLI commands — not generated by the LLM. LLMs compose the intent; PAC CLI executes it.

### 05 — Transparency & User Approval

**Pillar 5 — No surprises, ever**

The plugin must show every action it performs in real time and seek explicit user approval before any destructive, irreversible, or environment-altering operation. The user should never be surprised by what the plugin did.

## Pillar 1 — Engaging Loading Experience

### The Problem

When a skill generates a Dataverse schema or deploys a site, the terminal goes quiet for 30–120 seconds. The user doesn't know if it's working, stuck, or crashed. Research on perceived wait time shows that unexplained silence beyond 10 seconds causes users to assume failure — and they interrupt the process, making things worse.

### The Solution

Every skill and command must implement a **structured progress protocol** — a standardized way of reporting what stage the operation is in, what it just completed, and what's next.

**❌ Before — Current behavior**

User runs `/deploy-site`. Terminal shows "Running..." for 87 seconds. No output. User types "is it working?" and interrupts the process.

**✅ After — With progress protocol**

```
▸ Building site (step 1/4)         ✓
▸ Validating schema (step 2/4)     ✓
▸ Uploading to environment (step 3/4)  ◌ in progress...
▸ Activating site (step 4/4)       pending
```

### Implementation Pattern

Each skill defines its operation as a sequence of named stages. The skill emits stage transitions as structured console output that the agent can relay to the user.

```
# In every skill's SKILL.md — define the progress contract

## Progress Stages
Every long-running operation MUST print stage markers:

Stage format:  [STAGE n/N] Stage name... {status}
Status values: STARTED | DONE | FAILED | SKIPPED

# Example: Schema Generator deploying a schema
[STAGE 1/5] Validating schema definition... DONE (0.8s)
[STAGE 2/5] Checking for existing tables... DONE (2.1s)
[STAGE 3/5] Creating 4 tables in Dataverse... DONE (12.3s)
[STAGE 4/5] Configuring relationships... DONE (3.7s)
[STAGE 5/5] Generating table permissions... DONE (1.2s)
✓ Schema deployed: 4 tables, 23 columns, 6 relationships (20.1s total)
```

### What This Means for Each Phase

| Phase | Operations That Need Progress | Typical Duration |
|-------|-------------------------------|-----------------|
| **P1** | Schema push, permission deployment, Web API generation | 15–60s |
| **P2** | Accessibility scan (full site), SEO audit | 20–45s |
| **P3** | Server Logic deployment, Power Automate flow creation | 30–90s |
| **P4** | Power BI embed config, SharePoint connection setup | 20–60s |
| **P5** | Knowledge index build, AI model configuration | 30–120s |
| **P6** | Compliance scan, CI/CD pipeline generation, full site deploy | 45–180s |

> **Acceptance criterion:** No operation that takes longer than 10 seconds may ship without implementing the stage progress protocol. This is enforced in the PR checklist.

## Pillar 2 — Action Verification

### The Problem

The LLM says "I've created 4 tables in Dataverse" — but did it? The user has no way to verify without opening the maker portal and checking manually. This erodes trust. In regulated environments (the Sovereign Cloud use case), unverifiable actions are a non-starter.

### The Solution

Every mutating action must follow a **Do → Verify → Report** cycle. The agent performs the action, then runs an independent verification step to confirm it succeeded, then reports the verified result with evidence.

**❌ Before — Trust the LLM**

Agent: "I've created the Contact table with 8 columns and configured Read permissions for the Customer web role."

*User has no way to verify this without opening the maker portal.*

**✅ After — Verify and prove**

Agent: "I've created the Contact table. Let me verify..."

- `pac pages list` → confirms site exists
- `Dataverse query` → confirms 8 columns present
- `Permission check` → confirms Customer role has Read access

**✓ Verified:** Contact table (8 cols), Customer role (Read). [View in maker portal →]

### The Verification Pattern

```python
# Every skill that modifies state must implement this pattern:

## Verification Protocol

# Step 1: Execute the action
execute_action()      # e.g., create tables via Dataverse API

# Step 2: Independently verify the result
verify_result()       # e.g., query Dataverse to confirm tables exist

# Step 3: Report with evidence
report_with_evidence()
#   - What was requested
#   - What was done
#   - Verification result (pass/fail)
#   - Link to view in maker portal / admin center
#   - Rollback instructions if something is wrong
```

### Verification Methods by Phase

| Phase | Action | Verification Method |
|-------|--------|-------------------|
| **P1** | Schema creation | Query Dataverse metadata API to confirm tables, columns, and relationships match spec |
| **P1** | Table permissions | Read back permissions via Web API and compare against intended configuration |
| **P2** | Accessibility fixes | Re-run axe-core scan and confirm issue count reduced to zero |
| **P3** | Server Logic deploy | Invoke the endpoint with a test payload and confirm expected response |
| **P3** | Power Automate flow | Trigger a test run and verify flow execution history shows success |
| **P4** | Power BI embed | Fetch embed token and verify report renders (HTTP 200 from embed URL) |
| **P6** | Site deployment | `pac pages list` confirms site is active; HTTP GET to site URL returns 200 |
| **P6** | Compliance config | Scan response headers for CSP, HSTS, cookie flags; compare against template |

> **Acceptance criterion:** No skill that creates, modifies, or deletes Dataverse data, site configuration, or deployment state may ship without an automated verification step. The verification must use a different code path than the action itself (e.g., don't verify a create by reading the create response — query independently).

## Pillar 3 — Sub-Agent Architecture

### The Problem

Long-running operations (site deployment, full accessibility scan, schema migration) block the main conversation. The user can't ask questions, adjust parameters, or start a new task while waiting. Additionally, complex operations pollute the main context window with implementation details the user doesn't need to see.

### The Solution

Use Claude Code's native sub-agent system to offload long-running and context-heavy operations to isolated sub-agents. The main conversation stays responsive. The sub-agent runs with its own context, specific tool permissions, and reports back a clean summary when done.

### Sub-Agent Definitions for the Plugin

The plugin should ship with purpose-built sub-agents for each category of long-running work:

| Sub-Agent | Triggers On | Tools Allowed | Model | Reports Back |
|-----------|-------------|---------------|-------|-------------|
| **deploy-agent** | Site upload, code-site push, activation | `Bash(pac *), Read, Write` | sonnet | Deploy status, site URL, any errors |
| **schema-agent** | Dataverse schema creation, migration | `Bash(pac *), Read, Write, WebFetch` | sonnet | Tables created, columns, relationships, verification result |
| **audit-agent** | Accessibility scan, security audit, metadata validation, compliance check | `Read, Grep, Glob, Bash(npx axe-core *)` | sonnet | Issue count, severity breakdown, auto-fix suggestions |
| **integration-agent** | Power BI embed setup, SharePoint connection, Copilot Studio agent creation | `Bash(pac *), Read, Write, WebFetch` | sonnet | Integration status, config summary, verification result |
| **pipeline-agent** | CI/CD pipeline generation, environment migration | `Read, Write, Bash(git *), Bash(az *)` | sonnet | Pipeline file paths, test run results |

### Example Sub-Agent Definition

```markdown
# .claude/agents/deploy-agent.md
---
name: deploy-agent
description: Handles Power Pages site deployments, uploads,
  and activations as a background process.
tools: Bash, Read, Write
model: sonnet
---

# You are a Power Pages deployment specialist.

## Protocol
1. Print [STAGE n/N] markers for every step
2. Use `pac pages upload-code-site` for all uploads
3. After upload, verify with `pac pages list`
4. After activation, HTTP GET the site URL to confirm 200
5. Report back: site URL, deploy time, verification status
6. If any step fails, report the error and suggest recovery steps

## Constraints
- NEVER modify source code — only deploy what exists
- ALWAYS use PAC CLI for deterministic operations
- ALWAYS verify after every mutating action
```

### When to Delegate vs. Inline

| Operation | Expected Duration | Approach | Rationale |
|-----------|------------------|----------|-----------|
| Generate a React component | <5s | Inline (main agent) | Fast, context-relevant, user wants to see output |
| Deploy site to environment | 30–120s | **Sub-agent** | Long, deterministic, blocks conversation |
| Full accessibility audit | 20–60s | **Sub-agent** | Generates large output that pollutes main context |
| Create a single table | 5–15s | Inline with progress | Short enough to inline, but needs progress indicator |
| Schema migration (10+ tables) | 60–180s | **Sub-agent** | Complex, long, lots of intermediate output |
| Generate CI/CD pipeline files | <10s | Inline | File generation is fast; only deploy is long |
| Run CI/CD pipeline | 120–300s | **Sub-agent** | Very long; user should be free to work on other tasks |

> **Acceptance criterion:** Any operation expected to take longer than 30 seconds must be implemented as a sub-agent with its own context. The sub-agent must follow the progress protocol (Pillar 1) and verification protocol (Pillar 2).

## Pillar 4 — PAC CLI Determinism

### The Problem

LLMs are probabilistic. They are excellent at generating code, composing configurations, and reasoning about architecture. They are unreliable at executing the same API call identically every time. When an LLM constructs and executes a Dataverse REST API call directly, minor variations in URL encoding, header formatting, or parameter ordering can cause intermittent failures that are extremely hard to debug.

### The Principle

**LLMs compose. PAC CLI executes.** The LLM's job is to determine *what* needs to happen (intent). The PAC CLI's job is to *make it happen* (execution). This separation creates a deterministic execution layer that produces the same result every time, regardless of LLM temperature or token sampling.

**❌ Before — LLM as executor**

LLM constructs a raw HTTP POST to the Dataverse Web API with hand-assembled JSON body, manually constructed URL, and custom auth headers. Works 90% of the time. The other 10% fails silently or produces inconsistent results.

**✅ After — LLM as composer, PAC CLI as executor**

LLM determines: "Need to upload code site from ./build to 'Contoso Portal'."

Emits: `pac pages upload-code-site --rootPath "./src" --compiledPath "./build" --siteName "Contoso Portal"`

PAC CLI handles auth, URL construction, retries, and error formatting deterministically. Works 100% of the time.

### Determinism Classification

Every operation in the plugin should be classified as either **deterministic** (use PAC CLI or scripts) or **generative** (LLM-appropriate). This classification must be documented in the skill's SKILL.md.

| Operation Type | Classification | Execute With | Examples |
|---------------|---------------|-------------|----------|
| Site upload / download | ✅ Deterministic | `pac pages upload-code-site` / `pac pages download-code-site` | Deploy, download site content |
| Site listing / status | ✅ Deterministic | `pac pages list` | Check if site exists, get site ID |
| Auth profile management | ✅ Deterministic | `pac auth create` / `pac auth list` | Environment connection |
| Solution management | ✅ Deterministic | `pac solution export` / `pac solution import` | Environment promotion |
| Bootstrap migration | ✅ Deterministic | `pac pages bootstrap-migrate` | V3 to V5 migration |
| Code generation (React, TS) | 🔵 Generative | LLM | Components, services, config files |
| Schema design (table/column decisions) | 🔵 Generative | LLM | Deciding which tables, columns, relationships to create |
| Architecture recommendations | 🔵 Generative | LLM | Security model design, component architecture |
| Schema execution (creating tables) | 🟡 Hybrid | LLM composes → script executes | LLM generates the schema JSON; a deterministic script pushes it to Dataverse via API |
| Compliance header configuration | 🟡 Hybrid | LLM composes → template applies | LLM selects the template (GDPR, FedRAMP); a deterministic script applies it |

> **Acceptance criterion:** Every skill's SKILL.md must include a "Determinism Classification" section that lists each operation and whether it's deterministic, generative, or hybrid. Deterministic operations must use PAC CLI or pre-tested scripts. No skill may use LLM-generated raw API calls for operations that have a PAC CLI equivalent.

## Pillar 5 — Transparency & User Approval

### The Problem

An agent that silently creates 6 Dataverse tables, modifies security roles, and deploys to production — all from a single prompt — is terrifying in an enterprise context. Even if every action succeeds, the user didn't see what happened, didn't consent to each step, and can't explain to their team or compliance officer what the plugin did on their behalf. Opacity destroys trust faster than any bug.

### The Principle

**Show everything. Ask before anything irreversible.** The plugin must operate with radical transparency: every action is logged, every decision is explained, and every destructive or environment-altering operation requires explicit user confirmation before execution.

### The Three Transparency Layers

**Layer 1 — Action Log (always on)**

Every action the plugin performs must be printed to the console in real time. Not just the result — the action itself. The user should be able to read the log and reconstruct exactly what happened, in what order, and why.

**❌ Before — Black box**

```
Agent: "Done! Your Customer Service portal is ready."
```

*The user has no idea what was created, configured, or deployed.*

**✅ After — Full action log**

```
📋 Action Log:
  1. Created table: Contact (8 columns)
  2. Created table: Case (12 columns)
  3. Created table: Knowledge Article (6 columns)
  4. Set relationship: Contact → Case (1:N)
  5. Set relationship: Case → Knowledge Article (N:N)
  6. Created web role: Customer (scope: Contact)
  7. Created table permission: Contact (Read, Create) → Customer role
  8. Created table permission: Case (Read, Create) → Customer role
  9. Generated React components: ContactList, CaseForm, KBSearch
  10. Deployed to: https://contoso-portal.powerappsportals.com

✓ 10 actions completed. 0 errors.
```

**Layer 2 — Decision Explanation (for non-obvious choices)**

When the plugin makes a design decision that the user didn't explicitly request, it must explain *why*. This is especially important for security-related defaults.

```
ℹ️ Decision: Set Contact table permission to Read + Create only
   (not Update or Delete).
   Reason: Customer-facing portals should follow least-privilege.
   Customers can view their own contacts and create new ones,
   but cannot modify or delete existing records.
   Override: Add --permissions "Read,Create,Update,Delete" to
   grant full CRUD.
```

**Layer 3 — Approval Gates (for irreversible or high-impact actions)**

Certain actions must pause and ask for explicit user confirmation before proceeding. The plugin must never assume consent for destructive operations.

### Actions That Require Approval

| Action Category | Examples | Why Approval Is Needed |
|----------------|----------|----------------------|
| **Destructive operations** | Delete table, drop column, remove web role, clear site data | Data loss is irreversible |
| **Production deployment** | `pac pages upload-code-site` to a production environment | Affects live users immediately |
| **Security configuration** | Create or modify table permissions, change web role scope, modify CSP headers | Misconfiguration = data exposure |
| **Schema migration** | Add/remove columns on tables with existing data, change column types | May cause data truncation or loss |
| **External service connection** | Connect Power BI workspace, link SharePoint site, create Copilot Studio agent | Grants cross-service access |
| **Compliance-critical changes** | Modify GDPR/FedRAMP/HIPAA configuration, change cookie consent settings | Regulatory implications |

### The Approval Protocol

```
# Before any approval-gated action, the plugin must:

## Step 1: Announce intent
⚠️ I'm about to: Deploy site code to PRODUCTION environment
   Target: https://contoso-portal.powerappsportals.com
   Changes: 3 new components, 2 modified pages, 1 new Server Logic handler

## Step 2: Show impact summary
   Impact:
   - 3 new React components will be added (ContactList, CaseForm, KBSearch)
   - 2 existing pages will be overwritten (Home, Support)
   - 1 Server Logic handler will go live (case-validation)
   - Estimated downtime: none (hot deploy)

## Step 3: Request explicit approval
   Proceed? [Y/n]

## Step 4: Log the approval
   ✓ User approved deployment at 2026-03-15T14:32:07Z
```

### Transparency Levels by Context

Not every action needs the same level of transparency. The plugin should scale its transparency to the risk level of the operation:

| Context | Action Log | Decision Explanation | Approval Gate |
|---------|-----------|---------------------|--------------|
| **Local file generation** (components, configs) | ✅ Show files created | ❌ Not needed | ❌ Not needed |
| **Local analysis** (accessibility scan, SEO audit) | ✅ Show scan progress | ✅ Explain flagged issues | ❌ Not needed |
| **Dev environment changes** (schema push, permission config) | ✅ Full action log | ✅ Explain security defaults | ⚠️ Prompt for confirmation |
| **Production deployment** | ✅ Full action log | ✅ Explain all changes | ✅ **Mandatory approval** |
| **Destructive operations** (delete, drop, clear) | ✅ Full action log | ✅ Explain consequences | ✅ **Mandatory approval** |
| **Security / compliance changes** | ✅ Full action log | ✅ Explain regulatory impact | ✅ **Mandatory approval** |

### Undo & Rollback Visibility

When an action supports rollback, the plugin must tell the user how to undo it — immediately after the action completes, not buried in documentation.

```
✓ Deployed 3 components to production.

↩️ To rollback: Run /rollback-deploy --timestamp 2026-03-15T14:32:07Z
   This will restore the previous site state from the auto-backup
   created before deployment.
```

> **Acceptance criterion:** Every skill must implement Layer 1 (action log) for all operations. Skills that modify environment state must implement Layer 2 (decision explanation) for non-obvious defaults. Skills that perform destructive, production, security, or compliance operations must implement Layer 3 (approval gate). No approval-gated action may proceed without explicit user confirmation.
